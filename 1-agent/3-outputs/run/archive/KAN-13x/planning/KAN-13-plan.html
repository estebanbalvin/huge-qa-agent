QA Test Plan: Team Popularity Dashboard (KAN-13)

1. Analysis Summary

URL: https://futbolbuggy.geekqa.net/

Goal: To provide a real-time leaderboard showing which team has the most active fanbase. The dashboard must be visible on the main page and update immediately after a vote is cast.

2. Requirements & Acceptance Criteria

  • Ranking Visibility: A "Dashboard" or "Standings" section must be immediately visible.
  • Data Visualization: Team name, crest image, and vote count are mandatory.
  • Sorting: Teams must be in descending order by votes.
  • Real-Time Update: Vote count must update without a full page reload.
  • Responsive: Must work on mobile devices.
  • Capacity: Handle large vote numbers.

3. Risk Assessment

  • Real-time sync failure: The counter might not update until a manual refresh.
  • Incorrect sorting: The hierarchical order might not correctly re-rank after a vote.
  • Responsive issues: Large vote numbers or long team names might break the layout on mobile.
  • Data integrity: Votes might be lost or double-counted during high participation.

4. Proposed Test Types

  • Functional: Verify voting updates the counter and re-ranks teams.
  • Accessibility: WCAG 2.2 AA audit across mobile, tablet, and desktop breakpoints.
  • Performance: Lighthouse audit for load times and core web vitals.
  • Responsive: Testing at 375px, 768px, and 1440px.

5. Early Developer Checks (Suggestions)

  • Unit test the sorting logic (descending order).
  • Validate WebSocket or EventSource triggers for counter updates.
  • Verify team image aspect ratios and fallback states.

6. Selected Workflow Skills

The following skills will be executed in order:

  1. generate-test-cases - To create detailed scenarios.
  2. execute-tests - To run Playwright browser tests.
  3. run-accessibility-tests - For WCAG compliance.
  4. run-lighthouse-tests - For performance and SEO.
  5. log-results - To finalize the QA report.
